Spring 之旅

在诞生之初，创建 Spring 的主要目的是用来替代更加重量级的企业级 Java技术，尤其是 EJB（企业级 JavaBean 【Enterprise JavaBean，EJB】)。相对于 EJB 来说，Spring提供了更加轻量级和简单的编程模型。它增强了简单老式 Java 对象（Plain Old Java object，POJO）的功能，使其具备了之前只有 EJB 和其他企业级 Java 规范才能具有的功能。

随着时间推移，EJB 以及 Java 2 企业版（Java 2 Enterprise Edition，J2EE）在不断演化。EJB 自身也提供了面向简单 POJO 的编程模型。现在，EJB 也采用了依赖注入（Dependency Injection，DI）和面向切面编程（Aspect-Oriented Programming，AOP）的理念，这毫无疑问是受到 Spring 成功的启发。

简化 Java 开发

Spring 是一个开源框架，最早由 Rod Johnson 创建。

Spring 是为了解决企业级应用开发的复杂性而创建的，使用 Spring 可以让简单的 JavaBean 实现之前只有EJB 才能完成的事情。

Spring 是如何简化 Java 开发的？

为了降低 Java 开发的复杂性，Spring 采取了以下 4 种关键策略：

  • 基于 POJO 的轻量级和最小侵入性编程；
  • 通过依赖注入和面向接口实现松耦合；
  • 基于切面和惯例进行声明式编程；
  • 通过切面和模板减少样板式代码。

几乎 Spring 所做的任何事情都可以追溯到上述的一条或多条策略。

激发 POJO 的潜能

很多框架通过强迫应用继承它们的类或者实现它们的接口从而导致应用与框架绑死。

Spring 竭力避免因自身的 API 而弄乱你的应用代码。Spring 不会强迫你实现 Spring 规范的接口或继承 Spring 规范的类，相反，在基于 Spring 构建的应用中，它的类通常没有任何痕迹表明你使用了 Spring。最坏的场景是，一个类或许会使用 Spring 注解，但它依旧是 POJO。

尽管形式看起来很简单，但 POJO 一样可以具有魔力。Spring 赋予 POJO 魔力的方式之一就是通过 DI 来装配它们。

依赖注入 DI

依赖注入这个词让人望而生畏，现在已经演变成一项复杂的编程技巧或设计模式理念。但事实证明，依赖注入并不像它听上去那么复杂。在项目中应用 DI，你会发现你的代码会变得异常简单并且更容易理解和测试。

DI 功能是如何实现的

任何一个有实际意义的应用（比 Hello World 示例更复杂）都会由两个或者更多到类组成，这些类相互之间进行协作来完成特定的业务逻辑。按照传统的做法，每个对象负责管理与自己相互协作的对象（即它所依赖的对象）引用，这将会导致高度耦合和难以测试的代码。

举个例子，考虑下面程序所展现的 ZhangSan 类。

可以看到，ZhangSan 在它的构造函数中自行创建了 ChineseHamburger。这使得 ZhangSan 紧密地和 ChineseHamburger 耦合到了一起，因此极大地限制了 张三 的营业范围。如果一个人正好要吃肉夹馍，那么ZhangSan 能立马做出来。但是如果他想吃牛排，或者想吃一份炒河粉，那么 ZhangSan 就无能为力了。

更糟糕的是，为这个 ZhangSan 编写单元测试将出奇的困难。在这样的一个测试中，你必须保证当 ZhangSan 的 sell() 调用的时候，cHamburger 的 beEaten 方法也要被调用。但是没有一个简单明了的方式能够实现这一点。很遗憾，ZhangSan 将无法测试，肉夹馍店可能就要因此关门大吉了……

耦合具有两面性（Two-Headed-Beast）。

  • 一方面，紧密耦合的代码难以测试、难以复用、难以理解，并且典型的表现出 ”打地鼠“ 式的 Bug 特性（修复一个 Bug，将会出现一个或者一堆 Bug）。
  • 另一方面，一定程度的耦合又是必须的——完全没有耦合的代码什么也做不了。为了完成有实际意义的功能，不同的类必须以适当的方式进行交互。总而言之，耦合是必须的，但是应当被小心谨慎地管理。

通过 DI，对象的依赖关系将由系统中负责协调各对象的第三方组件在创建对象的时候进行设定。对象无需自行创建或管理它们的依赖关系。

传统对象的使用，对象要获取资源都是自己主动出击。

DI 依赖注入机制的引入，使对象变成被动，等待 DI 容器来创建并注入对象所需的资源。

这里的要点是 LiSi 没有与任何特定的 Food 实现发生耦合。对他来说，被要求制作的食物实现了 Food 接口，那么具体是哪种类型的食物就无关紧要了。这就是 DI 所带来的最大收益——松耦合。如果一个对象只通过接口（而不是具体实现或初始化过程）来表明依赖关系，那么这种依赖就能够在对象本身毫不知情的情况下，对不同的具体实现进行替换。
